## Component Inventory

### New Components

#### 1. TimingBar

**Location:** `prismlab/frontend/src/components/timeline/TimingBar.tsx`
**Parent:** ItemCard (rendered below the item image and gold cost)
**Purpose:** Horizontal segmented bar showing good/on-track/late timing zones

**Visual Specification:**
- Total width: 100% of ItemCard width (currently ~72px based on w-12 image + padding)
- Height: 6px
- Background: `surface-container` #201F1F
- Corner radius: 0px (DESIGN.md mandate)
- Three contiguous segments, widths proportional to time range of each zone
- Good segment: `radiant` #6AFF97
- On-track segment: `secondary-fixed-dim` #D4A017
- Late segment: `primary-container` #B22222
- Weak confidence: all segments rendered at 40% opacity
- Moderate confidence: all segments rendered at 70% opacity
- Strong confidence: all segments rendered at 100% opacity

**Interaction:**
- Hover: Shows tooltip with detailed timing data (see TimingTooltip below)
- No click action on the bar itself

**Props interface:**
```typescript
interface TimingBarProps {
  buckets: TimingBucketUI[];        // Processed timing data
  confidence: "strong" | "moderate" | "weak";
  isUrgent: boolean;                // Steep win-rate falloff detected
  isPurchased: boolean;             // Hide bar when item purchased
  currentGameClock: number | null;  // GSI game clock in seconds, null if no GSI
  currentGold: number | null;       // GSI gold, null if no GSI
  itemCost: number | null;          // Item gold cost for "X gold away"
}
```

#### 2. TimingTooltip

**Location:** Inline within TimingBar (rendered on hover)
**Purpose:** Shows detailed timing breakdown with win rates and confidence

**Visual Specification:**
- Background: `surface-container-highest` #353534
- Padding: 8px
- Position: above the timing bar, centered horizontally
- Width: 160px fixed
- No border (DESIGN.md no-line rule). Uses ambient glow shadow: `0 0 16px rgb(255 180 172 / 0.05)`
- Corner radius: 0px

**Content layout (top to bottom):**
1. Zone rows (one per zone):
   - Zone label ("Good", "On-track", "Late") -- 12px, weight 400, color matches zone
   - Time range ("< 20 min") -- 12px, weight 600, `on-surface`
   - Win rate ("58% WR") -- 12px, weight 600, color matches zone
2. Divider: 1px `outline-variant` #5A403E at 15% opacity (ghost border)
3. Confidence line: "Based on X games (strong)" -- 11px, weight 400, `on-surface-variant` at 60%

#### 3. UrgencyBorder (CSS animation, not a component)

**Applied to:** ItemCard `<button>` element when `isUrgent === true && !isPurchased`
**Purpose:** Visually separates timing-critical items from flexible ones (TIME-02)

**Visual Specification:**
- CSS class: `timing-urgent`
- Effect: Pulsing box-shadow using crimson glow
- Animation: `pulse-urgency 2s ease-in-out infinite`
- Keyframes:
  - 0%, 100%: `box-shadow: 0 0 0px rgba(178, 34, 34, 0)`
  - 50%: `box-shadow: 0 0 12px rgba(178, 34, 34, 0.15)`
- Subtle -- must not distract during gameplay (21-CONTEXT.md specifics)
- Respects `prefers-reduced-motion: reduce` -- animation disabled, static 1px `primary-container` border-left instead

#### 4. LiveTimingMarker (within TimingBar)

**Purpose:** Shows "you are here" position on timing bar during GSI-connected games (TIME-04)

**Visual Specification:**
- Rendered as absolute-positioned element inside TimingBar
- Width: 2px
- Height: 10px (extends 2px above and below the 6px bar)
- Color: `on-surface` #E5E2E1
- Position: left offset calculated as `(currentGameClock / maxTimeBucket) * 100%`, clamped to 0-100%
- When game clock exceeds max bucket time: marker sits at right edge

**"X gold away" text:**
- Rendered below the TimingBar (outside the bar element)
- Font: Manrope, 12px, weight 400
- Color: `on-surface-variant` #E2BEBA
- Format: "{gold_difference}g away" where `gold_difference = itemCost - currentGold`
- Hidden when: item is purchased, no GSI connection, or player has enough gold
- When player has enough gold: show "Affordable now" in `radiant` #6AFF97

#### 5. WindowPassedOverlay (within TimingBar)

**Purpose:** Visual indicator when game clock passes the "late" threshold (D-09)

**Visual Specification:**
- Entire TimingBar segments replaced with single fill: `surface-container-high` #2B2A29
- "Window passed" label appears centered below bar
- Font: Manrope, 11px, weight 600
- Color: `on-surface-variant` at 50% opacity
- Transition: 300ms ease opacity fade from active bar to passed state
- Item recommendation is NOT removed -- only urgency signal shifts

### Modified Components

#### ItemCard.tsx Modifications

**Current state:** Button with item image (48x48), gold cost label below. No timing data.

**Changes:**
1. Accept new prop: `timingData: TimingBarData | null` (null = no timing data for this item)
2. Accept new prop: `isUrgent: boolean` (from parent, computed from win-rate falloff)
3. Accept new props: `currentGameClock: number | null`, `currentGold: number | null` (from GSI store)
4. Render `<TimingBar>` below the gold cost span when `timingData !== null && !isPurchased`
5. Apply `timing-urgent` CSS class to outer button when `isUrgent && !isPurchased`
6. When purchased: hide TimingBar (D-10: purchased items don't show timing)

**Layout impact:** ItemCard height increases by approximately 14px (6px bar + 8px gap) when timing data is present. Width unchanged.

#### PhaseCard.tsx Modifications

**Changes:**
1. Accept timing data map and pass per-item timing data down to each ItemCard
2. Subscribe to GSI store for `game_clock` and `gold` values (via `useGsiStore` selectors)
3. Compute `isUrgent` per item and pass as prop

## Interaction States

### TimingBar States

| State | Condition | Visual |
|-------|-----------|--------|
| Default (no GSI) | `currentGameClock === null` | Three-zone bar, no marker, no gold text |
| Live tracking | GSI connected, item unpurchased | Three-zone bar + marker + "Xg away" |
| Affordable | GSI connected, `currentGold >= itemCost` | Bar + marker + "Affordable now" in green |
| Window passed | GSI connected, `gameClock > lateBucketMax` | Grey bar + "Window passed" label |
| Purchased | `isPurchased === true` | TimingBar hidden entirely |
| Weak confidence | `confidence === "weak"` | All bar segments at 40% opacity |
| Hover | Mouse over bar | TimingTooltip appears above bar |

### UrgencyBorder States

| State | Condition | Visual |
|-------|-----------|--------|
| Urgent, active | `isUrgent && !isPurchased && gameClock < lateThreshold` | Pulsing crimson glow |
| Urgent, passed | `isUrgent && !isPurchased && gameClock >= lateThreshold` | No glow (window passed state takes over) |
| Urgent, purchased | `isUrgent && isPurchased` | No glow |
| Not urgent | `!isUrgent` | No glow, no border change |
| Reduced motion | `prefers-reduced-motion: reduce` | Static 1px primary-container left border |

| Breakpoint | Behavior |

## Accessibility

| Requirement | Implementation |
|-------------|----------------|
| Timing bar color contrast | Green #6AFF97 on #201F1F = 10.2:1 ratio (AAA). Gold #D4A017 on #201F1F = 5.8:1 (AA). Crimson #B22222 on #201F1F = 2.7:1 -- supplemented by position (leftmost = good, rightmost = late) |
| Reduced motion | `@media (prefers-reduced-motion: reduce)` disables pulse animation, uses static border |
| Tooltip keyboard access | TimingBar receives `tabindex="0"`, tooltip shown on focus as well as hover |
| Screen reader | TimingBar has `role="img"` with `aria-label` describing timing range and urgency |
| Color-blind safety | Zones differentiated by position (left-to-right = good-to-late) and tooltip text, not color alone |

## Animation Budget

| Animation | Duration | Easing | Trigger |
|-----------|----------|--------|---------|
| Urgency pulse | 2s | ease-in-out | `isUrgent && !isPurchased` (infinite loop) |
| Tooltip appear | 150ms | ease-out | hover / focus on TimingBar |
| Tooltip disappear | 100ms | ease-in | mouse leave / blur |
| Window passed fade | 300ms | ease | game clock crosses late threshold |
| Marker position update | 0ms (instant) | none | GSI WebSocket tick (~1s interval) |

Total simultaneous animations: maximum 1 urgency pulse per item + 1 tooltip. No performance concern.

## Data Flow

### Backend to Frontend

Timing data flows through the existing recommendation response. The `ItemRecommendation` schema (or a sibling timing object on `RecommendResponse`) carries per-item timing benchmark data.

**New frontend type:**

```typescript
interface TimingBucketUI {
  time: number;          // seconds
  games: number;
  winRate: number;       // 0-1 float
  confidence: "strong" | "moderate" | "weak";
  zone: "good" | "ontrack" | "late";  // Pre-classified by backend
}

interface ItemTimingData {
  itemName: string;
  buckets: TimingBucketUI[];
  isUrgent: boolean;     // Backend-computed: steep win-rate falloff
  goodRange: string;     // e.g. "< 20 min" (pre-formatted for tooltip)
  ontrackRange: string;  // e.g. "20-25 min"
  lateRange: string;     // e.g. "> 25 min"
  goodWinRate: number;   // Aggregate for good zone
  lateWinRate: number;   // Aggregate for late zone
}
```

### GSI to TimingBar

Live game data flows through existing `useGsiStore`:
- `liveState.game_clock` -- seconds since game start
- `liveState.gold` -- current unreliable gold
- No new store needed. PhaseCard subscribes to GSI selectors and passes values as props to ItemCard/TimingBar.
