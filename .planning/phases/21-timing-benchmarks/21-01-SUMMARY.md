# Phase 21 Plan 01: Backend Timing Benchmark Pipeline Summary

Backend timing benchmark pipeline: zone classification utility consuming TimingBucket data, schema extension with ItemTimingResponse, context builder D-05 format output for Claude, and recommender post-LLM enrichment populating timing_data on RecommendResponse.

## What Was Built

### Task 1: Zone classification utility and schema extension (TDD)
- **timing_zones.py**: `classify_timing_zones()` partitions TimingBucket lists into good/ontrack/late zones using win-rate-relative thresholds (within 2% of peak = good, above weighted avg = ontrack, below = late)
- **Urgency detection**: `is_urgent` flag set when good-zone aggregate WR minus late-zone aggregate WR exceeds 10 percentage points
- **Time formatting**: Seconds-to-minutes conversion with zone-aware formatting (good: `< X min`, late: `> X min`, ontrack: `X-Y min`)
- **Confidence aggregation**: Weakest confidence across all buckets wins (weak > moderate > strong)
- **ItemTimingResponse**: New Pydantic model with pre-computed display-ready fields
- **timing_data field**: Added to RecommendResponse with `default_factory=list` for backward compatibility
- **LLM_OUTPUT_SCHEMA untouched**: Timing data is population-level statistics, never generated by Claude

### Task 2: Context builder timing section and recommender enrichment
- **_build_timing_section()**: Formats timing benchmarks per D-05 spec ("Black King Bar: good <15 min (58% WR), on-track 15-20 min, late >25 min (41% WR) [TIMING-CRITICAL]")
- **build() integration**: Timing section inserted after popularity section, before neutral catalog
- **_enrich_timing_data()**: Collects all recommended item_names from validated phases, looks up timing data in DataCache, classifies zones, constructs ItemTimingResponse objects
- **recommend() integration**: Enrichment step added after _validate_item_ids (Step 6b), timing_data passed to RecommendResponse constructor
