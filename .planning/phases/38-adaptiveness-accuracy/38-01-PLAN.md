<objective>
Implement diff-based re-evaluation context building so mid-game re-evals send only changed game state to Claude, reducing token usage by 40%+.

Purpose: Every mid-game re-evaluation currently rebuilds the entire context (hero info, matchup data, item catalog, popularity, timing benchmarks, team strategy, exemplars). Most of this is static within a game session. Diff-based context sends a compact summary of the unchanged portions plus only the fields that actually changed (new enemy items, lane result, damage profile, game clock, purchased items).

Output: Modified context_builder.py with build_diff(), modified recommender.py with snapshot tracking, token tracking on RecommendResponse.
</objective>

a) A frozen dataclass `EvalSnapshot` at module level (after the imports):
```python
from dataclasses import dataclass, field as dc_field

@dataclass(frozen=True)
class EvalSnapshot:
    """Snapshot of request fields for diff detection between evaluations."""
    hero_id: int
    role: int
    playstyle: str
    side: str
    lane: str
    lane_opponents: tuple[int, ...]
    allies: tuple[int, ...]
    all_opponents: tuple[int, ...]
    lane_result: str | None
    damage_profile: tuple[tuple[str, int], ...] | None  # sorted tuple of items
    enemy_items_spotted: tuple[str, ...]
    purchased_items: tuple[int, ...]
    enemy_context_hash: str  # hash of sorted enemy_context for change detection
    game_time_seconds: int | None
    turbo: bool
    full_context: str  # the full context string that was sent

    @classmethod
    def from_request(cls, req: RecommendRequest, full_context: str) -> "EvalSnapshot":
        import hashlib, json
        ec_hash = hashlib.md5(
            json.dumps([ec.model_dump() for ec in sorted(req.enemy_context, key=lambda e: e.hero_id)]).encode()
        ).hexdigest() if req.enemy_context else ""
        dp = tuple(sorted(req.damage_profile.items())) if req.damage_profile else None
        return cls(
            hero_id=req.hero_id, role=req.role, playstyle=req.playstyle,
            side=req.side, lane=req.lane,
            lane_opponents=tuple(sorted(req.lane_opponents)),
            allies=tuple(sorted(req.allies)),
            all_opponents=tuple(sorted(req.all_opponents)),
            lane_result=req.lane_result,
            damage_profile=dp,
            enemy_items_spotted=tuple(sorted(req.enemy_items_spotted)),
            purchased_items=tuple(sorted(req.purchased_items)),
            enemy_context_hash=ec_hash,
            game_time_seconds=req.game_time_seconds,
            turbo=req.turbo,
            full_context=full_context,
        )
```

b) A new method `build_diff()` on `ContextBuilder`:
```python
async def build_diff(
    self,
    request: RecommendRequest,
    rules_items: list[RuleResult],
    db: AsyncSession,
    prior: "EvalSnapshot",
) -> str:
```

This method detects which fields changed between `prior` and the current request by comparing:
- `lane_result` -- changed? Include "Lane result updated: {new_value}"
- `damage_profile` -- changed? Include new damage profile
- `enemy_items_spotted` -- new items (set difference)? List them
- `purchased_items` -- new purchases (set difference)? List them
- `enemy_context` -- hash changed? Include full new enemy context section
- `game_time_seconds` -- changed? Include new game clock section
- `lane_opponents` / `all_opponents` -- changed? If opponents changed, fall back to full build (new matchup data needed)
- `allies` -- changed? Fall back to full build

If opponents or allies changed, return `None` to signal "use full build instead."

The diff context format:
```
## Re-Evaluation Context (changes since last eval)

### Unchanged (retained from prior analysis)
Your hero: {hero_name} ({role_label}, {playstyle}), {side} {lane} lane
Opponents: {comma-separated opponent names}
{If allies exist: "Allies: {comma-separated ally names}"}

### What Changed
{Only include sections that actually changed:}
- Lane Result: {new value}
- New Enemy Items Spotted: {set difference items}
- New Purchases: {purchased items that are new since last eval, resolved to names}
- Damage Profile: {new profile}
- Game Clock: {new time}
- Enemy Team Status: {full section if hash changed}

### Already Recommended (from rules engine, do NOT duplicate)
{rules_lines -- always include since rules may have changed}

## Instructions
You previously recommended a build for this hero. The game state has evolved.
Re-evaluate ONLY phases affected by these changes. Keep unchanged phases stable.
Focus on explaining WHY the changes affect itemization. Cover ALL remaining phases.
```

The key sections omitted from diff: item catalog (~400 tokens), popularity (~200 tokens), pro reference (~200 tokens), timing benchmarks (~200 tokens), exemplars (~300 tokens), neutral catalog (~150 tokens), team strategy (unchanged). This saves ~1450 tokens per re-eval.

c) Add helper to detect if a request is a re-evaluation (has mid-game fields set):
```python
@staticmethod
def _is_reevaluation(request: RecommendRequest) -> bool:
    """True if request contains mid-game adaptation fields (not an initial draft eval)."""
    return (
        request.lane_result is not None
        or request.damage_profile is not None
        or len(request.enemy_items_spotted) > 0
        or len(request.purchased_items) > 0
        or (request.game_time_seconds is not None and request.game_time_seconds > 0)
    )
```

d) In `_deep_path()` and the LLM section of `recommend_stream()`, BEFORE calling `context_builder.build()`:
