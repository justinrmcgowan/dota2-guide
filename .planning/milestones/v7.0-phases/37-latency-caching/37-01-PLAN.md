<objective>
Replace the flat ResponseCache with a 3-tier HierarchicalCache that serves the best available cached response on partial input matches.

Purpose: Currently, changing a single opponent invalidates the entire cache. Hierarchical keys let us serve hero+role+lane builds instantly (L1, 1h TTL) even when the matchup-specific cache (L2, 5min) or exact-request cache (L3, 5min) misses.

Output: HierarchicalCache class in recommender.py, updated singleton in recommend.py, cache invalidation in refresh.py, unit tests.
</objective>

    Create `HierarchicalCache` class in recommender.py ABOVE the existing `ResponseCache` class (keep ResponseCache temporarily for reference, then remove it at end).

    Key design:
    - Three internal dicts: `_l1`, `_l2`, `_l3` each storing `dict[str, tuple[RecommendResponse, float]]`
    - L1 key: SHA256 of `{hero_id}:{role}:{lane}` -- TTL 3600s (1h)
    - L2 key: SHA256 of `{hero_id}:{role}:{lane}:{sorted_opponent_ids_joined_by_comma}` -- TTL 300s (5min)
      - `sorted_opponent_ids` = sorted(set(lane_opponents + all_opponents)) to normalize order
    - L3 key: SHA256 of full request JSON (model_dump_json(exclude_none=True)) -- TTL 300s (5min)
    - `get(request)` checks L3 first, then L2, then L1. Returns first hit or None.
    - `set(request, response)` writes to ALL THREE tiers (a full response is valid for all keys)
    - `set_l1(hero_id, role, lane, response)` -- direct L1 write for cache warming (no full request needed)
    - `cleanup()` evicts expired entries from all three dicts
    - `clear()` empties all three dicts
    - Constructor takes `l1_ttl=3600.0, l2_ttl=300.0, l3_ttl=300.0`
