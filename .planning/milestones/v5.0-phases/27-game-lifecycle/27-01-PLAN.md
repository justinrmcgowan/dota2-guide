<objective>
Add localStorage persistence to game and recommendation stores so state survives page refreshes, pipe match_id through the GSI pipeline from backend to frontend, and implement new-game detection that resets all match state when a different match begins.

Purpose: Without persistence, any page refresh or container restart loses all draft inputs, purchased items, and recommendations. Without new-game detection, stale data from the previous match leaks into the next one.

Output: Stores persist to localStorage via Zustand `persist` middleware. Match ID flows from GSI backend through WebSocket to gsiStore. useGameIntelligence detects match ID changes and triggers full match reset.
</objective>

**Frontend -- recommendationStore.ts:**
11. Wrap with Zustand `persist` middleware. The `purchasedItems` and `dismissedItems` fields are `Set<string>` which JSON.stringify converts to `{}`. Must add custom `storage` or `partialize`+serialization:
    ```typescript
    import { persist } from "zustand/middleware";

    export const useRecommendationStore = create<RecommendationStore>()(
      persist(
        (set, get) => ({
          // ... existing state and actions unchanged ...
        }),
        {
          name: "prismlab-recommendations",
          version: 1,
          storage: {
            getItem: (name) => {
              const raw = localStorage.getItem(name);
              if (!raw) return null;
              const parsed = JSON.parse(raw);
              // Rehydrate Set<string> from arrays
              if (parsed?.state?.purchasedItems) {
                parsed.state.purchasedItems = new Set(parsed.state.purchasedItems);
              }
              if (parsed?.state?.dismissedItems) {
                parsed.state.dismissedItems = new Set(parsed.state.dismissedItems);
              }
              return parsed;
            },
            setItem: (name, value) => {
              // Serialize Set<string> to arrays
              const clone = JSON.parse(JSON.stringify(value, (key, val) =>
                val instanceof Set ? [...val] : val
              ));
              localStorage.setItem(name, JSON.stringify(clone));
            },
            removeItem: (name) => localStorage.removeItem(name),
          },
        }
      )
    );
    ```
    This ensures `Set<string>` round-trips correctly through localStorage. Use Zustand v5's `createJSONStorage` pattern if cleaner, but the custom approach above is explicit and reliable.

```typescript
// --- 0. New game detection (D-05, D-06, D-07) ---
const matchId = live.match_id;
if (matchId && prevMatchIdRef.current && matchId !== prevMatchIdRef.current) {
  // Secondary confirmation: game state is hero selection (belt + suspenders)
  const isHeroSelection = live.game_state === "DOTA_GAMERULES_STATE_HERO_SELECTION";
  // Primary trigger alone is sufficient (match ID is unique per game),
  // but log when secondary doesn't confirm for debugging
  if (!isHeroSelection) {
    console.info(`New match detected (${prevMatchIdRef.current} -> ${matchId}) without hero selection state (state: ${live.game_state})`);
  }

  // Full match reset (D-08)
  useGameStore.getState().clear();
  useRecommendationStore.getState().clear();
  useRefreshStore.getState().resetCooldown();

  // Reset all refs to initial state
  prevHeroIdRef.current = 0;
  firedPhasesRef.current = new Set();
  laneAutoDetectedRef.current = false;
  cooldownEndRef.current = 0;
  queuedEventRef.current = null;
  prevStateRef.current = {
    deaths: 0,
    netWorthAtLastRefresh: 0,
    roshanState: "alive",
    radiantTowers: 11,
    direTowers: 11,
  };
}
// Always track the latest match ID
if (matchId) {
  prevMatchIdRef.current = matchId;
}
```

3. Place this BEFORE the hero detection block so that on a new game, the clear happens first, then the new hero gets detected and set in the same tick. This avoids a flash of stale state.

4. The `clear()` on gameStore resets all match state (per D-08). Settings like Steam ID, recommendation mode, volume, budget cap are in separate localStorage keys managed by the Settings panel (per D-09) and are never touched.

5. Note: D-07 says "both signals required" but match ID change is the definitive signal. The hero_selection state is secondary confirmation that we log but don't gate on -- because match ID could change during any game state transition (e.g., post-game to new game goes through multiple states). The primary + secondary approach avoids missed resets.
